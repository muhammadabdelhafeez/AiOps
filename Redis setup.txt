# Redis 7.2.5 Offline/Restricted Network Installation Guide on RHEL 10

## 1. Purpose

This document explains the full steps performed to install, run, validate, and troubleshoot **Redis 7.2.5** on a restricted **Red Hat Enterprise Linux 10** server.

The environment has:

- Restricted internet access
- Corporate SSL inspection
- Limited external repository access
- Default RHEL repositories only
- Redis package unavailable through `yum/dnf`

Because of these restrictions, Redis was installed from source using the official Redis download URL with SSL bypass for testing.

## 3. Initial Attempt Using yum/dnf

### Command used

```bash
sudo yum install redis
sudo systemctl enable redis
sudo systemctl start redis
```

### Result

```bash
No match for argument: redis
Error: Unable to find a match: redis
Failed to enable unit: Unit redis.service does not exist
Failed to start redis.service: Unit redis.service not found.
```

### Why this happened

Redis was not available in the enabled RHEL 10 repositories:

- BaseOS
- AppStream
- Microsoft RHEL repository

The `redis.service` also did not exist because Redis was never installed by the package manager.

### Key point

This was not a Redis runtime issue. The installation failed because the package was not found in the configured repositories.

## 5. Testing Internet URL Access

Before downloading Redis manually, we tested whether the server could reach the Redis download site.

### Commands used

```bash
curl -I https://redis.io
curl -I https://download.redis.io
curl -I https://repo.maven.apache.org
curl -I https://apache.maven.org
```

### Result example

```bash
curl: (60) SSL certificate problem: self-signed certificate in certificate chain
```

For Maven repository:

```bash
HTTP/2 200
server: cloudflare
```

For Apache Maven:

```bash
curl: (6) Could not resolve host: apache.maven.org
```

### Explanation

There were two different issues:

1. **SSL certificate issue**
   - The corporate firewall/proxy was doing SSL inspection.
   - The Linux server did not trust the corporate root CA certificate.
   - This caused `curl: (60)` certificate errors.

2. **DNS issue for some domains**
   - Some domains did not resolve from the server.
   - Example: `apache.maven.org` returned DNS resolution failure.

## 6. Confirming SSL Inspection Issue

### Command without SSL bypass

```bash
curl -I https://download.redis.io
```

Expected failure in this environment:

```bash
curl: (60) SSL certificate problem: self-signed certificate in certificate chain
```

### Command with SSL bypass

```bash
curl -k -I https://download.redis.io
```

Result:

```bash
HTTP/1.1 200 OK
```

### Why we used `-k`

The `-k` option tells curl to skip certificate validation.

This was used only as a temporary workaround to confirm that:

- DNS works for `download.redis.io`
- TCP port 443 is open
- The firewall allows the connection
- The real problem is certificate trust

### Important note

Using `-k` is acceptable for testing or temporary installation in a restricted environment, but it is not the recommended long-term production solution.

The proper fix is to import the corporate root CA certificate into the Linux trust store.

## 11. Run Redis in Background

### Command used

```bash
src/redis-server --daemonize yes
```

### Why we did this

By default, Redis runs in the foreground. If you close the terminal, the process may stop.

Using:

```bash
--daemonize yes
```

starts Redis as a background process.

### Real output received

```bash
WARNING Memory overcommit must be enabled!
```

### Important explanation

This warning does not mean Redis failed.

Redis started, but Linux kernel memory overcommit was not enabled. Redis recommends enabling it because background save or replication may fail if memory overcommit is disabled.

---

## 12. Fix Redis Memory Overcommit Warning

### Temporary fix without reboot

```bash
sysctl vm.overcommit_memory=1
```

### Permanent fix

```bash
echo "vm.overcommit_memory = 1" >> /etc/sysctl.conf
sysctl -p
```

### Verify

```bash
sysctl vm.overcommit_memory
```

Expected output:

```bash
vm.overcommit_memory = 1
```

### Why we did this

Redis may use background processes for persistence and replication.

If memory overcommit is disabled, Redis can fail during:

- RDB snapshot
- replication
- background save
- high memory pressure situations

Setting this value improves Redis stability.

## 19. Optional: Create a systemd Service

For production usage, Redis should be managed by `systemd` instead of running manually.

### Example service file

Create the file:

```bash
sudo nano /etc/systemd/system/redis.service
```

Paste the following content. Update the path if Redis was extracted somewhere else.

```ini
[Unit]
Description=Redis Server
After=network.target

[Service]
Type=simple
ExecStart=/root/redis-7.2.5/src/redis-server --daemonize no
ExecStop=/root/redis-7.2.5/src/redis-cli shutdown
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

### Enable and start service

```bash
sudo systemctl daemon-reload
sudo systemctl enable redis
sudo systemctl start redis
```

### Check status

```bash
sudo systemctl status redis
```

### Why we do this

Using systemd allows Redis to:

- start automatically after reboot
- restart automatically if it crashes
- be managed using standard Linux service commands

### Copy certificate to Linux

```bash
sudo cp /tmp/corp-root-ca.crt /etc/pki/ca-trust/source/anchors/
```

Redis installation from source is complete when all below are true:

- Redis source package downloaded successfully
- Redis compiled successfully using `make`
- Redis started in background

## 24. Recommended Next Steps for Production

For production or enterprise use, complete these additional steps:

1. Move Redis from temporary/root directory to a standard path such as:

```bash
/opt/redis/redis-7.2.5
```

2. Create a dedicated Redis user.

3. Create a proper Redis configuration file.

4. Set Redis password using `requirepass`.

5. Bind Redis to the required interface only.

6. Open firewall port `6379` only from approved application servers.

7. Configure systemd service.

8. Import corporate CA certificate to avoid future `curl -k` usage.

9. Document Redis server IP, port, owner, and purpose.

## 25. Example Architecture Usage for AIOps

Redis can be used in the AIOps platform as a fast in-memory cache.

Example use cases:

- Store alert fingerprints for deduplication
- Cache recent RCA correlation results
- Track repeated alerts within a time window
- Store temporary session/context data
- Reduce repeated database lookups

Example key design:

```bash
alert:dedup:<source>:<ci>:<alert_hash>
```

Example value:

```json
{
  "first_seen": "2026-06-28T11:00:00",
  "last_seen": "2026-06-28T11:05:00",
  "count": 15,
  "severity": "Critical"
}
```

TTL example:

```bash
src/redis-cli setex alert:dedup:scom:server01:cpu_high 900 "active"
```

This keeps the key for 900 seconds, then automatically expires it.
